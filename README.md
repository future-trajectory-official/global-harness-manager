# Global Harness Manager

このリポジトリは、AIエージェントの役割・スキル・およびプロジェクトの「権限境界」をシステムレベルで統制する、グローバルハーネスです。

## 思想と目的 (Harness Engineering)

本リポジトリは、以下の思想に基づいて構築されています。

1. **ポータビリティと衛生管理:**
   OSのユーザーディレクトリを汚染せず、スタンドアロンなツール（`gh`バイナリ等）と相対パスを用いて、クリーンな再現性を提供します。
2. **アイデンティティ（アカウント）の物理的隔離:**
   複数のプロジェクト・複数のGitHubアカウントが混在する環境において、設定ミスによる「誤プッシュ」を物理的に遮断します。
3. **グローバルスキルの供給:** 複数のプロジェクトから参照可能な共通の `Skills`、`Rules`、`Workflows`
   を提供します。

---

## 導入と利用方法 (Getting Started)

このリポジトリにおけるすべてのセットアップと制御は、単一のワークフローを通じて行われます。

1. Antigravity（エージェント）に対して、以下のように依頼してください。
   > 「`.agents/workflows/onboard-new-member.md` を実行して」
2. エージェントが事前チェックを行い、必要な設定ファイルの内容確認などを案内します。
3. 以降はエージェントが自動で下記の機能すべてを順次設定・同期します。

---

## ワークフローの内部挙動 (各機能の詳細)

ワークフローが実行される際、内部では以下のスキル群が呼び出され、プロジェクトの保護と環境構築が行われます。将来のメンテナンスのために備忘として記します。

### 1. ホスト基盤の初期構築 (`setup-harness-env`)

必要なCLIツール（スタンドアロン版 `gh` バイナリ等）をプラットフォームを汚さずに導入し、`~/.bashrc`
にパスを追加します。これを最初に行う最大の理由は、**今後のGitHubフローをAI自身に自律的に操作させるための基盤**を整えるためです。

### 2. マルチアカウントの管理と保護 (`manage-git-identity`)

設定ファイル（`identities.txt`）に基づき、アカウント固有の SSH 鍵を生成します。 **【🔴
ワークフロー経由での強制実行】** このワークフローでは、セキュリティの観点から既存の SSH
鍵を環境から退避させ、**常に新しい鍵ペアを新規生成して、ユーザーに GitHub
への再登録を**させます。万一のインシデント時やアカウント切り替え時に該当する鍵のみを無効化可能になり、安全性が高まります。

### 3. プロジェクトへのハーネス装着 (`attach-harness-to-project`)

通信経路となるリモート URL を `git@github.com-[アカウント名]:...` に強制書き換え、プロジェクトごとの
`user.name` / `user.email` を固定します。無関係なGitアイデンティティでの `git push`
は通信ごと拒否されます。

### 4. スキルのグローバル同期 (`publish-harness-rules`, `publish-harness-skills`)

設定に基づき、環境をまたいで共通化すべきルールやスキルを他プロジェクトへ同期配信します。

## ディレクトリ構成

- `scripts/`:
  プラットフォームを隔離し構築する制御エンジン。各スクリプトの内容はワークフローによって自動で呼び出されます。
- `config/`: 認証および環境設定を管理（実際の `.txt` や `.md` は Git
  で無視され、ローカルに保持されます）。
- `bin/`: ダウンロードされたスタンドアロンバイナリ等が格納されます。
- `.agents/`: エージェントに供給される専門ツール群（Skills / Rules / Workflows）。
- `.local/`: コミットされない個人的な未完了タスク（Task）やメモを安全に保存するための保護領域。
